## What is working

### Authentication

- Email/password login, rotating refresh cookies, CSRF protection, logout, forgot password, and reset password are connected to the API.
- Email registration uses `POST /api/v1/auth/register-email`, sends a one-time verification link, and blocks password login until verification.
- Google Identity Services sends an ID token to `POST /api/v1/auth/google`.
- The API verifies Google signature, issuer, expiry, audience, subject, and `email_verified` with Google's Node.js library.
- Google `sub`, not email, is persisted as the external identity key.
- A new Google user receives a User, owner Workspace, external identity, and initial rotating session transactionally.
- An existing Gmail or Google Workspace address may link automatically. A colliding third-party Google address without an `hd` claim is rejected and continues with password login.
- Google display name/avatar initialize new users but do not overwrite an existing ClickFlow profile.

See [Authentication](../engineering/AUTHENTICATION.md) and [Google authentication](../engineering/GOOGLE_AUTH.md).

### Workspace/Space integration

- `Space` maps to backend `Workspace`, `Folder` to `Project`, and `List` to `Section`.
- Workspace, Project, and Section navigation and core CRUD use the API.
- Lists can exist directly below a Space through the hidden root-project adapter.
- Space creation supports Public/Private metadata; owner-only settings and owner-only soft deletion are connected.
- Create Space/Folder/List and destructive confirmations use centered application dialogs.
- New Projects receive protected Open, In progress, and Complete system statuses.
- Custom statuses support color, rename/recolor, and safe deletion with task fallback to Open.
- List and Board views create tasks inline in the selected status.
- Documents, comments, activity, timers, attachments, analytics, templates, archive items, and settings have integration slices.

### Profile/dashboard/notifications

- Profile name/avatar changes are persisted through the API.
- Dashboard metrics and filters use API-backed data.
- Header notifications are derived from task activity and retain browser-local read state.

## Highest-priority unfinished work

### P0 - secure or remove the legacy registration endpoint

There are currently two public registration paths:

- `POST /api/v1/auth/register-email` creates an unverified email account and is used by the frontend.
- `POST /api/v1/auth/register` immediately creates a verified session.

`AuthController` is decorated with `@Public()`, so the legacy endpoint is not limited to a trusted bootstrap caller even though the authentication document describes it as a trusted integration/bootstrap flow. A caller can bypass email verification by calling it directly.

Recommended resolution:

1. Remove `POST /auth/register`, or make it call the email-verification registration flow.
2. If bootstrap registration is genuinely required, move it to a seed/CLI operation or protect it with a separate internal-only control; do not leave it public.
3. Update OpenAPI and `docs/engineering/AUTHENTICATION.md`.
4. Add an HTTP test proving a public caller cannot obtain a session without verifying email.

Relevant code:

- `apps/api/src/auth/auth.controller.ts`
- `apps/api/src/auth/auth.service.ts`
- `apps/web/src/features/auth/data/auth-api.ts`

### P0 - add real authenticated E2E coverage

`docs/delivery/PROGRESS_CHECKLIST.md` still has the authenticated PostgreSQL-backed E2E criterion unchecked. Add at least these scenarios:

1. Register email -> obtain the verification token from a test adapter -> verify -> login.
2. Reject password login before verification.
3. Google login with a stubbed verifier creates an account and Workspace.
4. Returning Google `sub` reuses the same user and creates a new session.
5. Login -> create Space -> create Folder/List -> create task -> reload -> data remains.
6. Cross-workspace IDs are rejected.

Do not call real Google or Gmail from CI. Replace the verifier/mail adapter in the test application.

### P1 - complete Space membership and sharing

The current Share and Invite UI is not a complete business flow:

- Backend only exposes `GET /workspaces/:workspaceId/members`.
- There is no invite/create membership endpoint, invitation acceptance, role update, member removal, leave-Space, or ownership transfer.
- The private-Space invite field explicitly states that it is waiting for the member invitation API.
- Share dialog buttons are presentation-only and do not load real members.

Implement and document:

1. Invite by normalized email and invitation expiry/revocation.
2. Accept invitation while authenticated; decide how email mismatch is handled.
3. Owner/Admin/Member role rules for the MVP.
4. Remove member, leave Space, and transfer ownership safeguards.
5. Real member list and pending invitations in the Share dialog.
6. Audit events for every membership mutation.

### P1 - define and enforce Public/Private semantics

The `private` flag is persisted, but access behavior is still membership-based for both values. Before adding more endpoints, write one stable rule for:

- Who can discover a Public Space.
- Whether a Public Space can be opened or joined from a link.
- Whether Private Spaces are visible only through membership/invitation.
- Whether changing Public -> Private removes anyone or only changes discovery.

Enforce the rule in repositories/guards, not only in the frontend.

### P1 - complete Space archive and duplication

- Space deletion is a soft archive, but there is no Workspace restore endpoint or restore UI.
- The productivity archive currently restores Project, Task, and Template, not Workspace.
- Duplicate Space currently reports that Workspace management is unavailable.
- Folder/List duplicate actions create shallow empty copies; decide whether product copy means shallow or deep.

### P2 - replace remaining browser prompts

Folder/List rename and some document actions still use `window.prompt`. Replace them with validated application dialogs so creation, editing, and deletion use one consistent UX.

## Local configuration

Use `.env.example` as the key list. Keep actual values only in ignored files.

API environment must include at minimum:

- `DATABASE_URL`
- `JWT_ACCESS_SECRET`
- `JWT_REFRESH_SECRET`
- `CORS_ORIGIN=http://localhost:3000`
- `WEB_URL=http://localhost:3000`
- `GOOGLE_CLIENT_ID`

For real mail delivery configure `EMAIL_HOST`, `EMAIL_PORT`, `EMAIL_USER`, `EMAIL_PASS`, and `EMAIL_FROM` (or the supported `SMTP_*` aliases).

Web environment in `apps/web/.env.local`:

```dotenv
NEXT_PUBLIC_API_URL=http://localhost:3001/api/v1
NEXT_PUBLIC_GOOGLE_CLIENT_ID=<same Google Web application client ID>
```

Google Cloud Console must list the exact local Authorized JavaScript origin:

```text
http://localhost:3000
```

Do not add a client secret to a `NEXT_PUBLIC_*` variable.
